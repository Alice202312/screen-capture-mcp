## What it does

- `take_screenshot` — AI grabs a screenshot, gets the file path back
- `start_recording` / `stop_recording` — AI controls screen recording
- Cross-platform: Windows 🍎 macOS ✅ Linux ✅

## Why MCP?

MCP is the emerging standard for AI tool interop. Build the server once, use it in every MCP client — Claude Desktop, Cursor, Windsurf, Goose, whatever. No per-client wiring.

## Example workflow

```
You: "Screenshot my browser and tell me if the form validation error is visible"
AI: → take_screenshot → "Yes, the red error text is visible below the email field, 
                               but it may be below the viewport on mobile resolutions"
```

## Show HN: Screen Capture MCP Server — give AI agents real-time screen vision

Hey HN,

I've been building AI agent workflows and kept hitting the same wall: my AI assistant can write code, but it can't **see** what the code produces. Every visual debugging session meant manually switching windows, taking screenshots, and pasting them back — a flow-breaking interruption that defeats the whole point of having an agent.

So I built **Screen Capture MCP Server** — an open-source MCP server that exposes screen capture (screenshots + video recording) as structured tools any MCP-compatible AI can call.

## The technical design

It's deliberately simple:

- **Protocol**: MCP (Model Context Protocol) — the same interface Claude Desktop, Cursor, Windsurf, and others use
- **Backend**: `ffmpeg` with platform-specific capture sources:
  - Windows: `-f gdigrab`
  - macOS: `-f avfoundation`
  - Linux: `-f x11grab`
- **Python 3.10+**, ~300 lines, no external deps except `mcp`
- **Auto dependency resolution**: checks for ffmpeg on startup (with install hints), auto-installs the `mcp` Python package silently

## The tool interface

```python
# Tools exposed to the AI:
take_screenshot(output_path)      # → image file path
start_recording(output_path)     # → "recording started"
stop_recording()                 # → video file path
get_recording_status()           # → bool
check_dependencies()             # → {ffmpeg, mcp, platform, python}
```

The AI calls these as normal tool invocations — it decides when to screenshot or record based on the user's request.

## Why not just use pyautogui or mss?

Those libraries grab raw pixel buffers — fine for a Python script, but:
1. They produce static images, no video streaming
2. They don't expose a protocol interface, so wiring them to AI agents requires custom code per tool
3. Every MCP client would need its own integration

By implementing MCP, this works everywhere that speaks MCP. One server, every AI tool.
